# AP 开发助理智能体 — 需求讨论记录

## 背景

在 `ap-dev-assistant` 项目中，基于现有的 Kmon 数据分析与 Badjs 修复能力，讨论并完善了 AP 开发助理智能体的完整需求文档。

## 讨论要点

### 1. 需求文档完善

基于原始的三条需求，扩展为完整的需求文档（`doc/requirement.md`），包含：

- **功能一：Badjs 自动分析与修复** — 8 个阶段的全流程闭环（错误分析 → 源码定位 → 相似 Bug 搜索 → 报告 → 修复 → Review → 提交 → 验证）
- **功能二：用户反馈智能处理** — 读取金山协作反馈群消息，自动分析并回复，确认问题后触发修复
- **功能三：辅助测试** — 针对修复代码自动生成测试方案和测试用例
- **功能四：辅助开发** — 通过 CLI 创建业务模块，生成完整目录结构和模板代码

### 2. 双运行模式设计

需求明确了两种运行模式：

| 模式 | 说明 |
|------|------|
| **Mode A（Cursor 注入）** | 注入到 wpsweb 工程，通过 Cursor Agent 驱动，无需配置 API Key |
| **Mode B（独立部署）** | 配置大模型 API Key 后独立运行，支持 CLI / HTTP / 定时任务 |

两种模式在 Skill 定义和工具实现上完全复用，仅在协议层和 AI 调用层做适配。

### 3. AP 定义修正

- **AP 即 outline**，是金山办公旗下的在线文字处理产品，不是整个金山办公的全部产品
- 去掉了错误的「All-Platform」全称
- presentation / et 等不属于 AP，是其他产品

### 4. 项目命名

- 项目更名为 **AP 开发助理智能体（AP Dev Agent）**
- 架构图、服务名等全文统一

### 5. 是否扩大为全能开发助理

讨论结论：**建议扩大，且集成在同一个项目中**，理由：

- 架构天然支持（Skill 模块化，增加能力只是多写 Skill）
- Cursor 的 `.cursor/` 机制要求一个工程一套 Skills，拆分多项目无法同时使用
- 各能力间有强关联（项目学习增强修复定位，反馈分析反哺需求优先级）

建议的扩展能力（按优先级）：

| 优先级 | 能力 |
|--------|------|
| P0 已完成 | badjs 修复、告警/异常/指标分析 |
| P1 当前需求 | 独立模式 Agent 引擎 + 用户反馈 + 辅助测试 |
| P2 低成本 | 项目学习（架构梳理、模块导读） |
| P3 中期 | 需求开发辅助（方案生成、脚手架） |
| P4 远期 | CR 辅助、新人指南 |

### 6. 辅助开发功能补充

新增功能四：通过 CLI `ap-agent create-module <模块名>` 创建业务模块：

- 交互式引导选择模块类型（UI 组件 / 业务逻辑 / 数据模型 / 完整功能模块）
- 自动扫描项目规范，生成符合 AP 风格的目录结构和模板代码
- 包含组件、数据模型、服务层、状态管理、国际化、测试等完整骨架
- 自动完成路由注册、导出文件更新等集成工作

### 7. 外部平台集成

工具层需对接的外部平台：

- **Kmon** — 前端监控数据（已完成）
- **Ones** — 缺陷任务管理
- **Fantasy** — 待确认具体用途
- **金山协作** — 反馈群消息读写
- **GitLab** — MR 创建与 Review

### 8. 远期愿景

AP 验证成功后，推广到金山办公全产品线：

- Phase 6 新增「全产品线推广」计划
- 将 AP 特有配置抽象为可配置项
- Skill 泛化为通用逻辑 + 项目配置
